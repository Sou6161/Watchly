# Deploying Watchly

Everything here is a thing **you** have to do — it needs your Render, Neon, Expo,
and store accounts.

## 1. Database (Neon)

Create a Postgres project. You need **two** connection strings from the dashboard.

**Pooled** (`...-pooler...`) → `DATABASE_URL`, used at runtime. Neon's pooler is
PgBouncer in *transaction* mode, which does not support prepared statements —
Prisma uses them, so you **must** append `?pgbouncer=true`, or queries start
failing once there's any concurrency. Add `connection_limit=1` too, so a free
instance can't exhaust Neon's connection quota:

```
postgresql://…-pooler.…neon.tech/watchly?sslmode=require&pgbouncer=true&connection_limit=1
```

**Direct** (unpooled) → `DIRECT_URL`. Migrations take advisory locks and run DDL,
which needs a real session and cannot go through a transaction pooler.

**You don't need to create the schema by hand.** Render runs
`prisma migrate deploy` as part of the build, which applies the committed
migration in `apps/api/prisma/migrations/`. This is deliberately *not* `db push`:
`migrate deploy` only moves forward and never drops data, whereas `db push` can
silently destroy a column to make the database match the schema. `db push` stays
a local-development tool.

(Verified: applying the migration to an empty database creates all four tables
and all five enums.)

## 2. API (Render)

`render.yaml` at the repo root is a blueprint — point Render at the repo and it
picks it up. Note it builds from the **repo root**, not `apps/api`: the API
imports `@watchly/shared` as a workspace package, so npm has to install from the
workspace root or the import won't resolve.

Set these in the dashboard (they're marked `sync: false`, i.e. never committed):

| Variable | Value |
| --- | --- |
| `DATABASE_URL` | Neon pooled string |
| `DIRECT_URL` | Neon direct string |
| `TMDB_API_KEY` | TMDB v4 read token or v3 key — both work |
| `CRON_SECRET` | `openssl rand -hex 24` |

`JWT_ACCESS_SECRET` and `JWT_REFRESH_SECRET` are generated by Render. **Rotating
either signs everyone out**, so generate once and leave them.

Node is pinned to 22 (`.node-version` + `engines`) — the code uses APIs that
aren't in Render's older default.

### What's already handled for you

- **Schema migration** runs at build, before the new instance takes traffic.
- **Graceful shutdown**: Render sends `SIGTERM` on every deploy. The server stops
  the sweep, closes sockets, drains in-flight requests, and disconnects Prisma.
  Without this, votes get cut off mid-write and Neon connections leak on every
  single deploy until the quota is gone.
- **Rate limiting**: 20 auth attempts per IP per 15 minutes, 120 req/min
  elsewhere. `/health` is exempt — throttling it would make Render think the
  instance is dead and cycle it. `trust proxy` is set, so limits key off the real
  client IP rather than Render's proxy (otherwise one attacker throttles
  everybody).

The API refuses to boot if the two JWT secrets are missing or identical — that's
deliberate, not a bug you need to work around.

### Keeping the instance warm (do this — the cold start is a product problem)

Render's free instance sleeps after ~15 minutes idle and takes **~50 seconds** to
wake. Watchly is an app people open at 9pm on a couch: the first person each
evening would stare at a spinner for the better part of a minute, at exactly the
moment the product is trying to feel effortless.

**Ping `/health` every 10 minutes — but only during the evening.**

| Setting | Value |
| --- | --- |
| Service | [cron-job.org](https://cron-job.org) (free; supports hour restrictions) |
| URL | `https://watchly-api.onrender.com/health` |
| Method | `GET` |
| Interval | Every 10 minutes |
| Hours | 16:00 – 01:00 IST only |

Ten minutes, because Render sleeps after fifteen. UptimeRobot's free tier (5-min
interval) works too.

**Why windowed, not 24/7.** Render's free tier bills *instance-hours*, and hours
accrue while the instance is awake. Keeping it up around the clock is ~730
hours/month against a ~750 quota — it technically fits, with no margin, and breaks
the moment you add a second free service. The evening window is ~270 hours/month
and leaves room. Nobody starts a movie night at 4am; a rare late-night user can
eat the cold start.

**This costs Neon nothing.** `/health` deliberately makes **no database query**
(check `src/index.ts` — it returns uptime and nothing else). Neon's free tier bills
compute hours and its compute autosuspends when idle, waking only on an actual
query. So the keep-warm ping keeps Render up without ever waking Neon. Your Neon
compute is spent on real users only.

That's also *why* the health check doesn't touch the DB: if it did, a Neon hiccup
would fail the health check, and Render would conclude the app was dead and cycle
the instance — taking the live sessions and sockets with it.

**Do NOT use GitHub Actions for the keep-warm ping.** It's the obvious choice and
it's the wrong one: Actions bills a **minimum of one minute per run**, so a 10-min
ping is ~4,300 billed minutes/month against a 2,000-minute free allowance on
private repos. (Actions is fine for the *nightly* catalog sync — that's one run a
day.)

If the app gets real traction, the Starter plan ($7/mo) never sleeps and this all
goes away.

> Render and Neon revise their free-tier limits fairly often. Check the current
> quotas before relying on the exact figures above — the reasoning holds either
> way.

## 3. Catalog

Populate the catalog once, then it refreshes nightly on its own.

```bash
# Locally, against the production DB:
DATABASE_URL="<pooled>" DIRECT_URL="<direct>" TMDB_API_KEY="<key>" \
  npm run sync -w @watchly/api
```

Takes ~15 minutes and caches ~4,000 titles (titles with no trailer or no
streaming provider are dropped — roughly half of what TMDB returns).

**Nightly refresh** runs via `.github/workflows/sync-catalog.yml`, which POSTs to
`/internal/sync-catalog`. Add two GitHub Actions secrets:

- `API_URL` → `https://watchly-api.onrender.com` (no trailing slash)
- `CRON_SECRET` → the same value you set on Render

It's an external trigger rather than an in-process cron because Render's free
instance is *asleep* at 3am — a `setInterval` would never fire. The HTTP request
both wakes it and starts the job.

Verify it works without waiting for the schedule: run the workflow manually
(`workflow_dispatch` is enabled).

## 4. Mobile

Set the production API in `apps/mobile/.env`:

```
EXPO_PUBLIC_API_URL="https://watchly-0hvm.onrender.com"
```

**`EXPO_PUBLIC_*` values are inlined into the bundle at build time, and Metro
caches the result.** Editing `.env` alone changes nothing — the old value stays
compiled in. Always clear the cache after touching it:

```bash
npx expo start --clear
npx expo export --clear ...
```

Get this wrong and you ship an app pointing at a LAN address: it works flawlessly
on your machine and for nobody else on earth. Verify before shipping by grepping
the built bundle for the URL you expect:

```bash
npx expo export --platform android --output-dir /tmp/b --clear
grep -a -c "onrender.com" /tmp/b/_expo/static/js/android/*.hbc   # want >= 1
```

### You need a development build, not Expo Go

Two things **cannot work in Expo Go**, and both are load-bearing:

- **Streaming deep links.** The `LSApplicationQueriesSchemes` (iOS) and `queries`
  (Android) entries in `app.json` are compiled into *your* app's manifest. In
  Expo Go your JS runs inside *their* binary, with *their* manifest — so
  `canOpenURL` returns false and every "Play on Netflix" falls through to a web
  search, even with Netflix installed. The spec is blunt that if the deep links
  don't work, the product fails on the punchline. **Test these on a real device
  before you ship.**
- **The app icon.** Expo Go shows its own.

```bash
npm install -g eas-cli
eas login
eas build:configure
eas build --profile development --platform android   # or ios
```

## 5. Stores

See [STORE.md](./STORE.md) for listing copy. You'll need:

- Apple Developer account ($99/yr) and Google Play ($25 one-off).
- Screenshots — take them from a real device, on the swipe and results screens.
- A privacy policy URL. You collect email, password hash, and vote history;
  that's enough to require one on both stores.
